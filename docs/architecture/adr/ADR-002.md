# ADR-002: Board + SQL デュアル永続化アーキテクチャの採用

- **日付**: 2026-03-04
- **ステータス**: 採用済み（Accepted）
- **決定者**: architect エージェント

---

## コンテキスト

エージェント間の状態共有と開発フローの進捗追跡のために、永続化層のアーキテクチャを検討した。
GitHub Copilot CLI には `sql` ツール（SQLiteベースのセッション内DB）が利用可能であり、
これをファイルベースの Board JSON とどう組み合わせるかが課題だった。

### 検討したアーキテクチャ案

**案1: Board JSON のみ**
- Feature の状態をすべて JSON ファイルに保存
- セッション跨ぎで永続化される
- クエリは JSON パースで対応

**案2: SQL のみ**
- セッション内 SQLite DB に全状態を保存
- 高速クエリが可能
- セッション終了で揮発する

**案3: Board JSON（永続的正本）+ SQL ミラー（揮発性クエリ層）（本採用）**
- Board JSON をすべての真実のソース（Source of Truth）とする
- SQL はセッション内の Board JSON コピーとして使用
- Board JSON 更新と同時に SQL を同期更新

---

## 決定

**案3: Board JSON + SQL ミラーのデュアル永続化を採用する。**

---

## 理由

### 1. セッション跨ぎの永続化が必要

複数の開発セッションにまたがる Feature 開発では、セッション終了後も状態が保持される必要がある。
SQLite は CLI のセッション単位で揮発するため、単独では永続化層として機能しない。

### 2. SQL の高速クエリによる開発効率向上

Board の JSON パースは繰り返し行うと冗長なコードが必要になる。
SQL ミラーを維持することで以下のようなクエリが即座に実行できる:

```sql
-- 次に実行すべき Gate を特定
SELECT name FROM gates WHERE status = 'not_reached' LIMIT 1;

-- 実行中のタスク一覧
SELECT id, title FROM todos WHERE status = 'in_progress';

-- Gate 評価履歴
SELECT * FROM board_history WHERE action = 'gate_evaluated' ORDER BY timestamp DESC;
```

### 3. コンパクション後の状態復元が容易

Copilot CLI のコンテキストウィンドウが圧縮（コンパクション）された場合でも、
Board JSON を `view` で再読み込みし SQL テーブルを再構築すれば、直前の状態を完全に復元できる。

### 4. バリデーション層としての活用

SQL クエリによる整合性検証が Board JSON の直接パースより簡潔に書ける:

```sql
-- flow_state が単一かどうか検証
SELECT COUNT(*) as state_count FROM board_state;
-- 期待値: 1

-- 孤立した Gate（対応する flow_state がない）の検出
SELECT g.name FROM gates g
LEFT JOIN flow_states fs ON g.required_state = fs.name
WHERE fs.name IS NULL;
```

---

## 実装上の制約

### 同期の原則

Board JSON と SQL ミラーは**常に同期して更新する**。
どちらか一方だけを更新する操作は禁止とする。

```
# 正しいパターン
1. Board JSON を更新（edit ツール）
2. SQL ミラーを同期（sql ツール）
3. バリデーションクエリを実行

# 禁止パターン
- Board JSON だけ更新してSQLを更新しない
- SQLだけ更新してBoard JSONを更新しない
```

### 信頼の優先順位

セッション再開時など、Board JSON と SQL ミラーが不整合になった場合は**Board JSON を正本**として SQL を再構築する。

---

## トレードオフ

| 観点 | JSON+SQL（採用） | JSON のみ | SQL のみ |
|---|---|---|---|
| セッション跨ぎ永続化 | ✅ JSON が担保 | ✅ | ❌ 揮発する |
| 高速クエリ | ✅ SQL が担保 | ❌ JSONパース | ✅ |
| 実装コスト | △ 両方を同期維持 | ✅ シンプル | ✅ シンプル |
| コンパクション耐性 | ✅ JSON から再構築可 | ✅ | ❌ 揮発 |
| バリデーション | ✅ SQL クエリで検証 | ❌ コード量が増える | ✅ |

---

## 影響

- `manage-board` スキルが JSON + SQL の同期更新手順を定義する
- `orchestrate-workflow` スキルは Board 操作時に常に両方を更新する
- セッション開始時の Board ロード手順（JSON → SQL インポート）が標準化される
