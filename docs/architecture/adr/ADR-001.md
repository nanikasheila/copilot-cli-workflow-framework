# ADR-001: Sub-agent型オーケストレーションの採用

- **日付**: 2026-03-04
- **ステータス**: 採用済み（Accepted）
- **決定者**: architect エージェント

---

## コンテキスト

本フレームワークは「要求分析 → 設計 → 実装 → テスト設計 → 独立検証 → レビュー → ドキュメント → PR」という多フェーズの開発ワークフローを自動化する。
このワークフローを実現するために、エージェント協調のアーキテクチャを検討した。

### 検討したアーキテクチャ案

**案1: Skill Chain型（逐次スキル連結）**
- 独立したスキルを固定パイプラインで直列連結
- 各スキルが独立ドメインのみを保持
- 例: `start-feature` → `analyze` → `plan` → `implement` → `review` → `submit`

**案2: Sub-agent型（本採用）**
- 1つの中核スキル（`orchestrate-workflow`）がオーケストレーターとして各エージェントを動的に spawn
- Board という共有状態を通じてフェーズ間でデータを受け渡す
- `task` ツールによる並列・逐次の動的制御

---

## 決定

**Sub-agent型（案2）を採用する。**

---

## 理由

### 1. Board による状態共有がフェーズ間のコンテキストロスを防ぐ

Skill Chain型では各スキルが独立しているため、前フェーズの成果物を次スキルに渡す機構が別途必要になる。
Sub-agent型では Board という単一の永続化層を通じて全フェーズが成果物を共有できる。

### 2. 動的分岐が必要なシナリオがある

- `architect` エスカレーション（構造変更が必要な場合のみ呼び出し）
- レビューNG時の実装フェーズへのループバック
- `experimental` vs `development` vs `stable` による Gate の動的調整

これらはコンパイル時に決定できない実行時分岐であり、固定パイプラインのSkill Chain型では対応が困難。

### 3. 読み取り専用エージェントの並列実行が安全に行える

`task` ツールは複数エージェントを同時 spawn できる。
Board への書き込みがないエージェント（analyst, impact-analyst, test-designer等）は
競合リスクなしに並列実行でき、開発サイクルを高速化できる。

### 4. エージェントの単体利用が可能

Sub-agent型はオーケストレーター経由が主だが、各エージェントは単体でも `/agent` コマンドや `task` ツールで直接呼び出せる。
フルフローが不要な場面（例: 単体のコードレビューだけ実行したい）にも対応できる。

---

## トレードオフ

| 観点 | Sub-agent型（採用） | Skill Chain型（不採用） |
|---|---|---|
| フェーズ間データ共有 | ✅ Board経由で一元管理 | ❌ スキル間で個別に受け渡し |
| 動的分岐 | ✅ 実行時に判断可能 | ❌ 固定パイプライン |
| 並列実行 | ✅ 読み取り専用エージェントを並列実行 | △ スキルが独立なら可能だが調整が複雑 |
| 複雑性 | △ Board管理・SQL同期の実装コスト | ✅ 各スキルが単純 |
| 単体利用 | ✅ エージェントを個別に呼び出せる | ✅ 各スキルが独立 |

---

## 影響

- `orchestrate-workflow` スキルがシステムの中核となる
- Board スキーマ（`board.schema.json`）の品質が全体の信頼性に直結する
- オーケストレーターは Board の `flow_state`・`gates`・`maturity` を管理する唯一の主体として設計する
